Obstacles & Solutions

Challenges & Learnings

Discovery Reliability

The Challenge

UDP Multicast often fails on multi-interface systems (e.g., VirtualBox, VPNs) because the OS routes packets to the wrong NIC.

The Fix

Implemented explicit interface binding. The app iterates through available interfaces to find the active LAN IP (e.g., 192.168.x.x) and binds the listener specifically to it.

Robust Socket I/O

The Challenge

TCP is a stream, not a message queue. conn.Read() might return partial chunks (e.g., 20KB instead of 64KB), corrupting file assembly.

The Fix

Switched to io.ReadFull() and a loop-based reader to guarantee the buffer is completely filled before writing to disk.

UI Performance

The Challenge

Sending a WebSocket event for every 64KB chunk (approx. 16,000 times/GB) flooded the browser event loop, causing the UI to freeze.

The Fix

Implemented throttling. The server tracks the last update time and only pushes progress events max once every 200ms.

Database Contention

The Challenge

Writing transfer progress to PostgreSQL after every chunk created massive disk I/O, slowing down the actual file write operation.

The Fix

In-Memory Buffer with Flush: Progress is tracked in a Go struct in memory and only flushed to the database on completion or pause.

Large File Handling

The Challenge

Attempting to read whole files into memory caused crashes with files > 2GB (RAM exhaustion).

The Fix

Strictly enforced streaming architecture using io.CopyBuffer. RAM usage is now constant (~64KB) regardless of file size.
